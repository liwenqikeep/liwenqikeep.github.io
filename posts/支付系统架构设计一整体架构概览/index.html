支付系统架构设计（一）：整体架构概览

前言

支付系统是互联网业务中最复杂、最核心的基础设施之一。它不像普通的业务系统——支付系统一旦出错，损失的是真金白银。正因如此，支付系统的架构设计需要极高的严谨性和完备性。

本文从一个架构师的视角，系统性地拆解一个完整支付系统的整体架构。我们将以支付宝、微信支付等一线支付平台的公开设计为参考，推演一个工业级支付系统的分层架构设计。

一、支付系统的复杂性来源

在设计支付系统之前，先理解它为什么复杂。

支付系统的核心挑战：

1. 资金安全 ──────────── 任何一笔资金差错都不可接受
2. 高可用 ────────────── 支付失败直接导致业务中断，可用性要求 99.99%+
3. 一致性 ────────────── 分布式环境下，必须保证资金账目最终一致
4. 合规性 ────────────── 不同国家/地区的金融监管要求
5. 多渠道 ────────────── 对接微信、支付宝、银联、银行卡等 N 个渠道
6. 高并发 ────────────── 大促期间峰值 QPS 可达数十万

一个设计良好的支付系统，核心目标可以总结为四个字：不丢、不错。

三、各层详细设计

3.1 接入层（Gateway Layer）

接入层是支付系统的门面，负责统一收敛所有外部请求。

接入层的功能：
┌──────────────────────────────────────────────┐
│              接入层职责矩阵                      │
├──────────────────────────────────────────────┤
│ 1. 统一认证             签名验签、API Key 校验 │
│ 2. 限流熔断             按商户/接口/QPS 限流   │
│ 3. 参数校验             请求参数合法性校验       │
│ 4. 路由转发             路由到核心业务层        │
│ 5. 安全防护             IP 黑白名单、防重放攻击  │
│ 6. 协议转换             XML ↔ JSON、加密解密    │
└──────────────────────────────────────────────┘

关键设计要点：

  • 签名机制：所有接入请求必须携带签名，防止篡改和重放攻击
  • 接口版本化：/v1/pay、/v2/pay，支持多版本共存和平滑升级
  • 统一异常处理：定义标准的错误码体系，所有异常统一格式返回
// 支付接口统一响应体
public class PaymentResponse<T> {
    private String code;        // 业务码：SUCCESS / PARAM_ERROR / SYSTEM_ERROR
    private String message;     // 提示信息
    private String requestId;   // 全局唯一请求 ID，用于链路追踪
    private T data;             // 业务数据

    public static <T> PaymentResponse<T> success(T data) {
        return new PaymentResponse<>("SUCCESS", "成功", null, data);
    }

    public static <T> PaymentResponse<T> fail(String code, String message) {
        return new PaymentResponse<>(code, message, null, null);
    }
}

3.2 核心业务层（Core Layer）

核心业务层是支付系统的大脑，负责所有支付相关业务流程的编排。

                          ┌───────────┐
                          │  支付请求   │
                          └─────┬─────┘
                                │
                          ┌─────▼─────┐
                          │  参数校验   │
                          └─────┬─────┘
                                │
                          ┌─────▼─────┐
                          │  风控检查   │  ← 风控系统
                          └─────┬─────┘
                                │
                          ┌─────▼─────┐
                          │  创建支付单  │  ← 入库，状态 INIT
                          └─────┬─────┘
                                │
                    ┌───────────┴───────────┐
                    │                       │
              ┌─────▼─────┐          ┌─────▼─────┐
              │ 渠道路由决策  │          │  调用渠道   │
              └─────┬─────┘          └─────┬─────┘
                    │                       │
                    └───────────┬───────────┘
                                │
                          ┌─────▼─────┐
                          │  处理结果   │
                          │ 成功 → 记账  │
                          │ 失败 → 退款  │
                          └───────────┘

核心领域对象：

领域对象 说明 状态
PaymentOrder 支付订单，用户侧视角 INIT → PAYING → SUCCESS / FAIL
PaymentTransaction 支付流水，渠道侧视角，一次支付可能有多次流水 INIT → CALLING → SUCCESS / FAIL
RefundOrder 退款订单 INIT → PROCESSING → SUCCESS / FAIL

核心业务层最关键的设计是状态机，这将在本系列的第三篇文章中深入展开。

3.3 账务层（Accounting Layer）

账务层负责资金账目的记录和核对。这是支付系统区别于普通业务系统的核心所在。

账务层的核心模型——复式记账（Double-Entry）：

  用户 A 支付 100 元给商户 B：

  借：用户 A 的账户    -100 元
  贷：商户 B 的账户    +100 元

  ── 任何一笔交易，借方总额 = 贷方总额
  ── 任何时候，所有账户余额之和 = 0（有正有负）

关键设计要点：

  • 记账与支付分离：支付成功先记账流水，再异步更新余额，通过最终一致性保证
  • 流水不可篡改：流水表采用 append-only 设计，不允许 UPDATE 和 DELETE
  • 对账闭环：每日对账（内部对账 + 渠道对账），确保资金准确性

3.4 渠道层（Channel Layer）

渠道层是支付系统的四肢，负责与外部支付渠道交互。

渠道适配器接口（抽象）：

┌──────────────────────────────────────────┐
│            ChannelAdapter              │
├──────────────────────────────────────────┤
│ + pay(PayRequest): PayResponse           │
│ + refund(RefundRequest): RefundResponse  │
│ + query(QueryRequest): QueryResponse     │
│ + parseNotify(Map): NotifyResult         │
└──────────────┬───────────────────────────┘
               │ 实现
     ┌─────────┼────────────┬──────────────┐
     │         │            │              │
┌────▼────┐ ┌───▼────┐ ┌───▼────┐ ┌──────▼──────┐
│微信适配器 │ │支付宝  │ │银联    │ │银行卡       │
│          │ │适配器  │ │适配器  │ │适配器        │
└──────────┘ └────────┘ └────────┘ └─────────────┘

关键设计要点：

  • 适配器模式：每个渠道实现统一接口，新增渠道不影响核心逻辑
  • 异步通知处理：渠道的回调通知必须做幂等处理，防止重复入账
  • 超时重试：支付请求超时后，先查单（query）再决定是否重试

关键设计决策：同步 vs 异步

环节 方式 原因
用户 → 接入层 同步（HTTP） 用户体验需要实时反馈
接入层 → 核心层 同步（RPC） 需要直接返回收银台信息
核心层 → 渠道层 同步（HTTP） 调起渠道收银台，需要即时的跳转 URL
账务入账 异步（MQ） 入账不需要阻塞支付链路
渠道回调 异步（MQ） 渠道异步通知，经过 MQ 削峰
对账 异步（定时任务） 每日 T+1 对账即可

六、技术选型与考量

技术组件 选型 选型理由
基础框架 Spring Boot 3.x 生态成熟，社区活跃
RPC Dubbo / Spring Cloud OpenFeign 微服务间调用
网关 Spring Cloud Gateway 统一接入、路由、限流
消息队列 Apache Kafka / RocketMQ 高吞吐、可靠性、事务消息
分布式事务 Seata（TCC 模式） 支付核心链路强一致性
配置中心 Nacos 配置动态刷新
数据库 MySQL + ShardingSphere 数据量大时需要分库分表
缓存 Redis 幂等去重、热点数据
链路追踪 SkyWalking 全链路监控

选型原则：在支付系统中，稳定 > 性能 > 功能。优先选择经过大规模验证的成熟技术。

七、架构设计核心原则

7.1 防腐层（Anti-Corruption Layer）

外部渠道的变化不应该影响核心业务逻辑。

渠道层作为防腐层，封装所有外部系统的复杂性。即使某个渠道的接口发生重大变更，只需要修改对应的适配器实现。

7.2 依赖倒置

核心业务层不依赖渠道层和账务层的具体实现，只依赖接口。

payment-core
   ├── repository.PaymentRepository (接口)  ← 由 infrastructure 实现
   ├── client.ChannelClient (接口)          ← 由 channel 层实现
   └── client.AccountClient (接口)          ← 由 account 层实现

7.3 最终一致性优先

除了资金操作本身，其他环节都优先考虑最终一致性而非强一致性。

支付链路中，真正需要强一致性的只有记账操作。其他环节（通知商户、发送短信、更新统计）都可以通过异步来实现最终一致。

八、总结

本文从架构师视角，系统性地拆解了一个支付系统的整体分层架构。核心要点：

  1. 四层架构：接入层、核心业务层、账务层、渠道层，各层职责清晰
  2. 核心链路：支付请求经过"验签 → 参数校验 → 风控 → 创建单 → 渠道路由 → 调起渠道 → 异步通知 → 入账"的完整链路
  3. 关键设计：领域对象（PaymentOrder/PaymentTransaction/RefundOrder）、防腐层、依赖倒置、最终一致性
